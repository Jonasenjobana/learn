# MySQL 知识点总结

## 一、字段类型

### 1. 数值类型

| 类型 | 大小 | 范围（有符号） | 说明 |
|------|------|----------------|------|
| `TINYINT` | 1 byte | -128 ~ 127 | 小整数，`BOOLEAN` 的底层类型 |
| `SMALLINT` | 2 bytes | -32,768 ~ 32,767 | 小范围整数 |
| `MEDIUMINT` | 3 bytes | -8,388,608 ~ 8,388,607 | 中范围整数 |
| `INT` / `INTEGER` | 4 bytes | -2^31 ~ 2^31-1 | 最常用整数 |
| `BIGINT` | 8 bytes | -2^63 ~ 2^63-1 | 大整数，自增ID常用 |
| `FLOAT(M,D)` | 4 bytes | 单精度浮点数 | 不精确，避免用于金额 |
| `DOUBLE(M,D)` | 8 bytes | 双精度浮点数 | 不精确，避免用于金额 |
| `DECIMAL(M,D)` | 变长 | 精确小数 | **金额必用**，不会精度丢失 |
| `BIT(M)` | 1~8 bytes | 1~64 位 | 位字段 |

> **金钱用 DECIMAL，千万别用 FLOAT/DOUBLE！**

### 2. 字符串类型

| 类型 | 最大长度 | 说明 |
|------|----------|------|
| `CHAR(N)` | 255 字符 | 定长，性能好，适合固长字段（手机号、MMSI、MD5） |
| `VARCHAR(N)` | 65,535 bytes | 变长，最常用字符串类型 |
| `TINYTEXT` | 255 bytes | 短文本 |
| `TEXT` | 65KB | 长文本（文章正文等） |
| `MEDIUMTEXT` | 16MB | 较大文本 |
| `LONGTEXT` | 4GB | 超大文本 |
| `BINARY(N)` | 255 bytes | 类似 CHAR，存二进制 |
| `VARBINARY(N)` | 65KB | 类似 VARCHAR，存二进制 |
| `BLOB` | 65KB | 二进制大对象 |
| `MEDIUMBLOB` | 16MB | 中等二进制对象 |
| `LONGBLOB` | 4GB | 超大二进制对象 |
| `ENUM('a','b','c')` | 1~2 bytes | 枚举单选 |
| `SET('a','b','c')` | 1~8 bytes | 集合多选 |

> **VARCHAR vs TEXT**：VARCHAR 可加完整索引、有默认值；TEXT 必须前缀索引、不能有默认值。能用 VARCHAR 就别用 TEXT。

### 3. 日期和时间类型

| 类型 | 大小 | 范围 | 格式 | 说明 |
|------|------|------|------|------|
| `DATE` | 3 bytes | 1000-01-01 ~ 9999-12-31 | `YYYY-MM-DD` | 仅日期 |
| `TIME` | 3 bytes | -838:59:59 ~ 838:59:59 | `HH:MM:SS` | 时间/持续时间 |
| `DATETIME` | 8 bytes | 1000-01-01 ~ 9999-12-31 | `YYYY-MM-DD HH:MM:SS` | 最常用日期+时间 |
| `TIMESTAMP` | 4 bytes | 1970 ~ 2038-01-19 | `YYYY-MM-DD HH:MM:SS` | 带时区，可自动更新 |
| `YEAR` | 1 byte | 1901 ~ 2155 | `YYYY` | 年份 |

> **DATETIME vs TIMESTAMP**：TIMESTAMP(4B) 比 DATETIME(8B) 省空间，但只能表示到 2038 年。

### 4. JSON 类型（MySQL 5.7+）

| 类型 | 说明 |
|------|------|
| `JSON` | 自动校验 JSON 格式，可用 `JSON_EXTRACT()`、`->`、`->>` 操作 |

### 5. 空间数据类型（GIS）

`POINT`、`LINESTRING`、`POLYGON`、`GEOMETRY` 等，配合 `SPATIAL INDEX` 做空间查询。

## 二、ENGINE（存储引擎）

### InnoDB（默认，推荐）

```sql
ENGINE=InnoDB
```

**建表都应使用 InnoDB**，它是 MySQL 5.5+ 默认引擎，唯一支持事务和外键：

| 特性 | InnoDB |
|------|--------|
| 事务 | ✅ BEGIN/COMMIT/ROLLBACK |
| 行级锁 | ✅ 高并发下只锁受影响的行 |
| 外键约束 | ✅ 保证数据完整性 |
| 崩溃恢复 | ✅ redo log 保证崩了能恢复 |
| 聚簇索引 | ✅ 数据和索引一起存，主键查询极快 |

### 其他引擎（了解）

| 引擎 | 特点 | 适用场景 |
|------|------|----------|
| `MyISAM` | 表级锁，不支持事务 | 只读报表、数据仓库 |
| `MEMORY` | 数据在内存，重启丢光 | 临时表、缓存 |
| `CSV` | 数据存为 CSV 文件 | 数据交换 |
| `ARCHIVE` | 高压缩，只支持 INSERT+SELECT | 日志归档 |

## 三、CHARSET（字符集）

### 最大的坑：`utf8` ≠ 真·UTF-8

```sql
CHARSET=utf8      -- ❌ 最多 3 字节，存不了 emoji！
CHARSET=utf8mb4   -- ✅ 最多 4 字节，才是真正的 UTF-8
```

| 编码 | 字节数 | 能存什么 | 不能存什么 |
|------|--------|----------|------------|
| `utf8` (utf8mb3) | 1~3 bytes | 大部分汉字、英文 | emoji、特殊符号 |
| `utf8mb4` | 1~4 bytes | **所有 Unicode** | — |

### 排序规则 COLLATE

```sql
CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
```

| 排序规则 | 说明 |
|----------|------|
| `_unicode_ci` | 基于 Unicode 排序，准确，推荐 |
| `_general_ci` | 旧版规则，较快但不精确 |
| `_bin` | 按二进制比较，区分大小写 |

### 建表最佳实践

```sql
CREATE TABLE tbl_example (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='示例表';
```

### 注意事项总结

1. **最后一个字段不要有逗号** — 语法错误
2. `updated_at` 要加 `ON UPDATE CURRENT_TIMESTAMP` — 否则不会自动更新
3. **定长数据用 CHAR 不用 VARCHAR** — MMSI、手机号等
4. **唯一标识加 UNIQUE 索引** — 如 MMSI
5. **CHARSET 一律用 utf8mb4** — 忘记 utf8 的存在
6. **ENGINE 一律用 InnoDB** — 不要碰 MyISAM
7. **金额用 DECIMAL** — 精度是生命线

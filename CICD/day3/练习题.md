# Day3 练习题：部署、Secrets 与环境保护

这一天专门练安全部署相关能力。

## 题目 1：手动触发部署

创建一个 workflow：

```yaml
on:
  workflow_dispatch:
```

要求：

- 能在 Actions 页面手动点击运行。
- 不要先配置 push 自动部署生产。

## 题目 2：添加普通变量

在 GitHub 仓库中添加变量：

```text
DEPLOY_ENV=staging
```

要求：

- 在 workflow 中通过 `vars.DEPLOY_ENV` 读取。
- 在日志中输出它。

## 题目 3：添加 Secret

添加一个 Secret：

```text
DEPLOY_TOKEN
```

要求：

- 不要把真实 token 写进代码。
- 在 workflow 中使用 `secrets.DEPLOY_TOKEN`。
- 不要明文打印完整 secret。

## 题目 4：配置 environment

创建一个 GitHub Environment：

```text
staging
```

要求：

- 在 job 中写 `environment: staging`。
- 运行 workflow 并观察页面展示。

## 题目 5：生产审批

创建 `production` Environment。

要求：

- 配置 required reviewers。
- 修改 workflow 使用 `environment: production`。
- 观察部署前是否需要审批。

## 题目 6：最小权限

给 workflow 添加：

```yaml
permissions:
  contents: read
```

要求：

- 说明为什么要控制权限。
- 思考什么时候需要更高权限。

## 题目 7：部署前构建

在部署 job 中加入：

- checkout
- setup-node
- npm ci
- npm run build

要求：

- 只有构建成功才执行部署步骤。

## 题目 8：模拟部署

不连接真实服务器，只输出：

```text
deploy success
```

要求：

- 先跑通流程。
- 再考虑接入真实服务器或云平台。

## 题目 9：区分 staging 和 production

添加一个输入参数：

```yaml
workflow_dispatch:
  inputs:
    environment:
      type: choice
      options:
        - staging
        - production
```

要求：

- 根据输入选择部署环境。
- 思考 production 为什么需要更严格保护。

## 题目 10：综合小练习

写一个部署 workflow：

- 手动触发。
- 可选择 staging/production。
- 使用 secrets。
- 使用 environment。
- 构建成功后模拟部署。
- 设置最小 permissions。

## 检查标准

如果你做完了，至少应该能清楚解释：

- 为什么部署不应该一开始就自动上生产。
- `secrets` 和 `vars` 的区别。
- Environment 能保护什么。
- `permissions` 为什么要最小化。
- 一个安全部署 workflow 的基本结构。

# AI 学习文档生成规范

这个仓库用于整理编程、前端、后端、AI、运维、CI/CD 等学习资料。  
后续 AI 在本仓库生成学习文档时，应优先沿用现有 `day1`、`day2`、`day3` 这种阶段式风格，不要随意另起一套目录结构。

## 总体原则

- 始终使用简体中文。
- 面向有一定 JavaScript / 前端基础的学习者，解释时可以适当做技术对比。
- 内容要偏实战，不只写概念，要能让学习者跟着文件运行、修改、练习。
- 每个主题优先按 `day1`、`day2`、`day3` 递进生成。
- 每个 day 都应有明确主题，例如 `Day1：基础配置与入门练习`。
- 不要把所有内容堆在一个超长文件里，优先拆成阶段目录。
- 不要随意删除已有文件；修改已有文档时做增量修改。
- 如果已有索引文件，例如主题目录下的 `README.md`，应更新索引，不要重复创建功能相同的新入口。

## 推荐目录结构

主题目录建议这样组织：

```text
主题/
├─ README.md
├─ day1/
│  ├─ README.md
│  ├─ 练习题.md
│  └─ 示例代码
├─ day2/
│  ├─ README.md
│  ├─ 练习题.md
│  └─ 示例代码
└─ day3/
   ├─ README.md
   ├─ 练习题.md
   └─ 示例代码
```

如果是工程化主题，例如 webpack、Node、前端框架，可以包含：

```text
dayN/
├─ README.md
├─ 练习题.md
├─ package.json
├─ 配置文件
├─ public/
└─ src/
```

如果是 Python、算法、脚本类主题，可以包含：

```text
dayN/
├─ README.md
├─ 练习题.md
├─ index.py
└─ 其他示例文件
```

## 主题 README 写法

每个主题目录下的 `README.md` 应作为总入口，推荐包含：

- 主题说明：这套内容学什么，适合谁。
- 当前学习入口：列出已有 day。
- 阶段目录表：包含阶段、主题、入口。
- 阶段总结：总结每个 day 学了什么、重点是什么、学完能做什么。
- 学习目标：学完这些 day 后应该能做什么。
- 后续进阶方向：下一步可以学什么。

示例：

```md
# Webpack 入门到工程化

这套内容用于从零学习 webpack，并逐步过渡到真实项目构建能力。

## 当前学习入口

- [Day1：基础配置与资源引入](./day1/)
- [Day2：静态资源与生产构建](./day2/)
- [Day3：Babel 与代码分割](./day3/)
```

主题父级目录不能只放 day 链接，还应补充每个 day 的总结，方便后续 AI 和学习者快速判断当前进度。

推荐写法：

```md
## 阶段总结

| 阶段 | 主题 | 学完后应掌握 | 入口 |
| --- | --- | --- | --- |
| Day1 | 基础配置 | 跑通最小示例，理解核心概念 | [day1](./day1/) |
| Day2 | 常用能力 | 掌握真实项目常见配置和练习方式 | [day2](./day2/) |

### Day1 总结：基础配置

Day1 的目标是先把项目跑起来。

重点内容：

- 核心概念 1
- 核心概念 2

适合目标：

- 能完成目标 1
- 能完成目标 2
```

## 单个 Day 的 README 写法

每个 `dayN/README.md` 建议按下面结构写：

```md
# DayN：当天主题

这一阶段用于说明今天学什么，以及它和前一天的关系。

## 学习目标

学完 DayN 后，你应该能：

- 目标 1
- 目标 2
- 目标 3

## 文件说明

- [练习题](./练习题.md)
- [示例代码](./index.js)
- [配置文件](./webpack.config.js)

## 推荐学习方式

1. 先看 README 的重点提醒。
2. 阅读示例代码或配置。
3. 自己做练习题。
4. 运行示例，观察输出。
5. 修改关键配置，再观察变化。

## DayN 重点提醒

### 1. 重点概念

解释概念，并给简短示例。

## 建议练习

- 练习 1
- 练习 2
- 练习 3

## 入口命令

```bash
npm install
npm run dev
npm run build
```
```

注意：不同技术栈的命令应按实际项目调整，不要机械套用。

## 练习题写法

每个 day 推荐配一个 `练习题.md`，结构如下：

```md
# DayN 练习题：当天主题

这一天专门练某个能力，不重复前一天已经掌握的基础内容。

## 题目 1：题目名称

题目说明。

要求：

- 要求 1
- 要求 2
- 要求 3

## 题目 2：题目名称

题目说明。

## 检查标准

如果你做完了，至少应该能清楚解释：

- 检查点 1
- 检查点 2
- 检查点 3
```

练习题应从简单到综合，最后最好有一个综合小练习。

## 示例代码写法

生成示例代码时：

- 示例要能运行。
- 文件名、目录名要和 README 中的链接一致。
- 不要只写空壳，应包含当天核心知识点。
- 代码注释要少而清楚，只解释容易混淆的地方。
- 前端工程示例应包含 `package.json` 和必要配置文件。
- Python 示例应尽量包含可直接运行的 `index.py`。

## 阶段主题递进建议

生成新 day 时，要先看已有 day，避免重复。

推荐递进方式：

```text
Day1：基础入门，跑通最小示例
Day2：补齐常用能力，处理真实输入/资源/配置
Day3：进入工程化或进阶语法
Day4：错误处理、调试、测试或性能优化
Day5：综合项目或真实业务场景
```

不同主题可以调整，但要保持“由浅入深、当天目标明确”。

## 更新索引规则

新增 `dayN` 后，应同步更新：

- 主题目录下的 `README.md`
- 如果存在总览文档，也要更新总览
- 根目录 README 一般不需要每次更新具体 day 列表，除非目录规范发生变化

例如新增 `前端/webpack/day4` 后，应更新：

```text
前端/webpack/README.md
```

## 依赖管理规则

如果需要新增第三方依赖：

- 先确认当前项目使用 npm、pnpm 还是 yarn。
- 沿用已有包管理器。
- 可以生成 `package.json`，但不要擅自执行安装命令。
- 需要安装依赖时，应先征得用户同意。
- 优先选择成熟、常用、社区验证充分的库。

## 文档风格要求

- 标题清楚，章节短小。
- 多使用列表、表格、代码块帮助学习者扫描。
- 少写空泛介绍，多写“怎么做、为什么、如何验证”。
- 重点提醒要能帮助学习者避坑。
- 练习要可操作，避免只有“了解一下”。
- 示例命令和路径要尽量贴近 Windows 环境。
